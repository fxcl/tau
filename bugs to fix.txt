In normal Tau CLI sessions, most specialized tool definitions are "deferred" to optimize system startup and reduce initial context load. The CLI only
  exposes a small core of essential tools (like Read, Write, Edit, Bash, Glob, Grep) at the very beginning. Whenever the agent needs a more specialized tool
   (like WebSearch, WebFetch, TaskCreate, NotebookEdit, etc.), it must call ToolSearch first to fetch that tool's full JSON schema and register/load it
  dynamically into the session.

  Because ToolSearch itself is a helper system tool that manages the loading of other tools, it is typically loaded implicitly or not listed as a standard
  user-facing tool in the default session schema. When a test suite or table checklist runs, it looks for ToolSearch in the initially exposed user tool
  definitions. Since it isn't listed there as a typical calling tool, the automated checks flag it as "Not listed".
